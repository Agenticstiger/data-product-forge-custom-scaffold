# Build a YAML/Jinja bundle from scratch

**Time:** 15 minutes | **Difficulty:** Beginner | **Prerequisites:** [Getting Started](../getting-started/README.md)

A YAML/Jinja bundle is the easiest way to ship a custom scaffold — no Python code, just a manifest + Jinja templates. The engine's built-in `TemplatedCustomScaffold` does the rest.

## What you'll build

A bundle named `org-ci-bundle` that generates:

- `README.md` — derived from contract identity
- `.gitlab-ci.yml` — one deploy job per declared environment (Jinja `{% for %}` inline)

## Step 1 — Skeleton

```bash
mkdir org-ci-bundle && cd org-ci-bundle
mkdir templates
```

## Step 2 — Manifest

```yaml
# fluid-scaffold.yaml
apiVersion: fluid.dev/custom-scaffold.v1
bundle:
  name: org-ci-bundle
  version: 1.0.0
  description: My organisation's standard CI scaffold.
  author: Platform Team

patterns:
  - name: gitlab-ci
    description: GitLab CI + README from a fluid contract.
    supportedProductTypes: [SDP, ADP, CDP]

    variables:
      $schema: http://json-schema.org/draft-07/schema#
      type: object
      properties:
        ciStage:
          type: string
          default: deploy

    requiredContractFields:
      - metadata.owner.email
      - environments
      - environments.*.metadata.labels["cloud.accountId"]

    templates:
      - from: templates/README.md.j2
        to: README.md

      - from: templates/.gitlab-ci.yml.j2
        to: .gitlab-ci.yml
```

## Step 3 — Templates

### `templates/README.md.j2`

```jinja
# {{ product_name or product_id }}

{{ description }}

**Owner:** {{ owner.email }}

## Environments

{% for env in environments|sort -%}
- `{{ env }}`
{% endfor %}
```

### `templates/.gitlab-ci.yml.j2`

```jinja
# Auto-generated CI for {{ product_id }}
stages:
  - validate
  - deploy

validate:
  stage: validate
  script:
    - fluid validate

{% for env in environments|sort %}
deploy:{{ env }}:
  stage: {{ ciStage }}
  script:
    - fluid apply --env {{ env }}
{%- if env == "prod" %}
  when: manual
{%- endif %}
  only:
    - main
{% endfor %}
```

That's it. No `forEach:` in the manifest — Jinja's `{% for %}` does the work inline.

### ⚠ Use `.get()` for any optional contract field

The engine renders templates with Jinja's **`StrictUndefined`** mode, which
raises an error the moment a template touches a missing key. This is
deliberate — silent fallbacks to empty strings hide bugs. But it means you
must use `.get()` for any field the contract isn't *required* to declare.

```jinja
{# ❌ Crashes when column has no `labels` block #}
{% if column.labels.primaryKey %}...{% endif %}

{# ✅ Safe — returns None for missing keys #}
{% if column.get('labels', {}).get('primaryKey') %}...{% endif %}

{# ✅ Or use the `default` filter #}
{{ environments.dev.metadata.labels['cloud.region'] | default('eu-central-1') }}
```

If a field is *required* by your bundle (no sensible default), list it
under `requiredContractFields:` in the manifest — the engine will then
fail with a clear "missing required contract field X" error *before*
any template renders, instead of a confusing Jinja stack trace from
inside a template.

### Render-context cheatsheet

When your template renders, these top-level names are bound:

| Name | Source |
|---|---|
| `fluid` | full contract dict (escape hatch) |
| `metadata` | `contract["metadata"]` |
| `product_id`, `product_name`, `product_type` | shortcuts from `ContractHelper` |
| `owner`, `tags`, `labels` | shortcuts |
| `exposes`, `consumes`, `builds`, `environments` | parsed lists / dicts |
| `sovereignty`, `security`, `access_policy` | optional contract blocks |
| `bundle` | the bundle's own identity (`name`, `version`, `description`, `author`, `pattern_name`) |
| `<variable>` | anything declared in `patterns[].variables` in the consumer's contract |

`bundle.name` and `bundle.version` are useful for "Auto-generated by X v.Y" headers.

## Step 4 — Test it locally

```bash
cd ..
mkdir my-test && cd my-test
cat > contract.fluid.yaml <<'YAML'
fluidVersion: "0.7.4"
kind: DataProduct
id: my-test
name: My Test Product
description: Testing the org-ci-bundle.
metadata:
  owner: {team: platform, email: platform@example.com}
environments:
  dev:    {metadata: {labels: {"cloud.accountId": "111"}}}
  prod:   {metadata: {labels: {"cloud.accountId": "333"}}}

extensions:
  customScaffold:
    libraries:
      - id: ci
        source: { kind: path, path: "../org-ci-bundle" }
    patterns:
      - use: ci:gitlab-ci
        variables:
          ciStage: "build-and-deploy"
YAML

fluid custom-scaffold
```

You'll see:

```
✓ 2 files written, 0 failed
  README.md
  .gitlab-ci.yml
```

## Step 5 — Add static assets

If your bundle ships any files that shouldn't be Jinja-rendered (binary fixtures, sample data, pre-rendered docs), drop them into a top-level `static/` directory:

```
org-ci-bundle/
├── fluid-scaffold.yaml
├── templates/
│   ├── README.md.j2
│   └── .gitlab-ci.yml.j2
└── static/
    ├── docs/
    │   └── runbook.md           → copied to <output>/docs/runbook.md
    └── data/
        └── sample.csv           → copied to <output>/data/sample.csv
```

The engine copies every file under `static/` byte-for-byte, preserving paths. No declaration needed in the manifest.

## Step 6 — Determinism

```bash
mkdir /tmp/first-run && cp -r * /tmp/first-run/
rm README.md .gitlab-ci.yml
fluid custom-scaffold
diff -r . /tmp/first-run/
# (no output — byte-identical)
```

The bundle is deterministic by construction. Same inputs → same bytes.

## Step 7 — Share

Push to a git repo:

```bash
cd ../org-ci-bundle
git init
git add .
git commit -m "Initial org CI bundle"
git tag v1.0.0
git push origin main --tags
```

Anyone can now use it without `pip install`:

```yaml
extensions:
  customScaffold:
    libraries:
      - id: ci
        source:
          kind: git
          url: "https://github.com/example/org-ci-bundle"
          ref: "v1.0.0"
    patterns:
      - use: ci:gitlab-ci
```

The engine clones into the user's cache and uses it from there.
